## 1. 任务目标

本系统第一阶段解决的是：“在给定局面下，某个动作或某个效果处理声明是否在规则内合法，并输出原因”的裁定问题。

系统不负责寻找最优操作，不负责完整对局模拟，也不负责卡组构筑。

### 1.1 `task_type` 区分

当前 `task_type` 分为两种：

- `operation_legality_judgment`：判断发动、召唤、攻击宣言、选择对象等“操作”是否合法。`pre_state` 通常处于开放状态、连锁构建状态，或尚未进入当前效果处理。
- `effect_resolution_judgment`：判断效果处理是否能按声明方式适用。`pre_state` 通常处于连锁处理中，依赖 `state_timing` 与已经实际处理完成的 `resolution_history`。

当前 50 条 gold cases 的分布为：

- `operation_legality_judgment`：37 条；
- `effect_resolution_judgment`：13 条。

## 2. 第一阶段覆盖范围

优先覆盖的问题类型：

- 某效果是否能发动、是否满足发动条件；
- 某效果或卡的发动能否连锁当前连锁；
- cost 是否可以支付；
- 对象是否可以选择，以及处理时对象是否合法；
- 当前步骤、伤害步骤或召唤成功窗口是否允许该动作；
- 效果处理是否能按声明方式适用；
- 是否受到一回合一次限制；
- 是否受到持续效果、召唤限制、攻击限制、区域限制等外部约束。

第一阶段优先覆盖卡片与主题包括：

- 灰流丽、增殖的 G、无限泡影；
- 效果遮蒙者、PSY 骨架装备 γ；
- 阿不思的落胤、烙印与圣女相关卡；
- I:P 百变莱娜、S:P 小夜骑士；
- 以及官方 Q&A 中能覆盖发动窗口、连锁速度、cost、对象、伤害步骤、召唤限制、一次限制、持续效果和效果处理的代表性场景。

## 3. 输出定义

输出标签固定为四类：

- `legal`：当前操作或声明处理合法；
- `illegal`：当前操作或声明处理不合法；
- `depends`：信息不足，结论取决于缺失条件；
- `invalid_question`：问题不属于当前裁定任务边界。

不得再使用 `movement_correct` 或 `movement_incorrect`。

输出必须包含：

- 结论；
- 判断链条；
- 缺失信息；
- 使用的卡片文本或裁定依据；
- 不确定性说明。

其中 `depends` 必须配合非空的 `gold_answer.missing_info`。不确定性统一通过 `label: "depends"` 与 `missing_info` 表达，不另设自由命名的不确定性字段。

## 4. 判断流程

系统按以下逻辑顺序判断：

1. 是否存在当前操作或处理声明的合法窗口；
2. 候选动作是否允许在该窗口发动、宣言或适用；
3. 该动作是否符合发动条件、召唤条件、攻击条件或处理条件；
4. 若需要 cost，cost 是否能在当前规则约束下支付；
5. 若取对象，对象在发动时与处理时是否合法；
6. 是否受到一回合一次、同名限制或选项限制；
7. 是否受到持续效果、召唤限制、攻击限制、区域限制、伤害步骤限制等外部约束；
8. 对于效果处理类 case，结合 `resolution_history` 判断此前已经实际处理完成的连锁结果；
9. 输出结论、关键依据和缺失信息。

## 5. 评测标准

每个样例评测：

- 最终结论是否正确；
- 关键条件是否识别正确；
- 是否正确指出缺失信息；
- 是否引用了正确依据；
- 是否出现无依据裁定；
- 来源是否可追溯到官方 Q&A、官方规则书或官方卡片文本。

## 6. 第一阶段状态与下一阶段目标

第一阶段“提供 50 条手工 gold cases”的成功标准已经达成。

下一阶段质量目标是：

- 对现有 50 条 case 做逐条官方证据复核；
- 硬化 `official_ruling.source_updated_at`、日文/简中官方卡片文本覆盖、镜像一致性等机器校验；
- 当关键卡无 KONAMI 简中官方正文时，以本地 `cards.cdb`/`cards.db` 的 `secondary_reference` 提供中文卡文覆盖（`authority: local_cards_cdb`），不得伪装为 `official_card_text`；
- 建立候选集合、待复核集合和 RAG 检索评测集；
- 在合适时引入 CI，自动运行 `python check_jsonlschema.py --self-test`；
- 后续扩充新 case 前，优先保证 50 条基线数据的稳定性和可追溯性。
